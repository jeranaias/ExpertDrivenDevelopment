Why This Matters

Gemini 2.5 on GenAI.mil is a capable model, but it requires extreme specificity to produce working Power Platform code. Without priming, it will:

  • Confuse Power Fx with JavaScript — generating forEach, arrow functions, and try/catch inside Power Apps formulas
  • Hallucinate functions — inventing functions like VLookUp(), Split(), or CALCULATE() that do not exist in Power Fx
  • Ignore GCC High constraints — suggesting commercial endpoints, third-party connectors, and AI Builder features that are unavailable in your environment
  • Skip delegation warnings — writing formulas that silently return incomplete data from large SharePoint lists

The solution is two-fold:

  1. The RTFC pattern (Role, Task, Format, Constraints) — a structured prompting framework that gives Gemini the context it needs on every request
  2. Meta-prompting — instructing the AI to ask you clarifying questions before writing any code, eliminating the guesswork that causes hallucinated control names and column names

GenAI.mil's Agent Designer (launched March 2026) lets you save these instructions permanently so every conversation starts correctly. This guide covers both options: a quick-paste primer for immediate use, and the full Agent Designer configuration for long-term use.

The Quick-Paste Primer (Option A — No Agent Designer)

If you don't have Agent Designer access yet, or you need to work in a standard GenAI.mil chat session, paste the following at the start of every session. This is the condensed version (~470 words) that covers the essential rules.

Session-Only

You lose this primer when the session ends. You must paste it again at the start of every new conversation. For a permanent solution, use Option B (Agent Designer).

Agent Designer Setup (Option B — Recommended)

Availability Note

Agent Designer launched on GenAI.mil on March 10, 2026. If you don't see it yet, your tenant may not have it enabled. Use Option A (quick-paste) until it's available.

Agent Designer lets you create a persistent AI assistant with baked-in instructions. Every conversation with your agent starts from the same foundation — no re-pasting required.

  1. Log into GenAI.mil

    Navigate to gemini.genai.mil and sign in with your CAC/credentials. You should see “Gemini Enterprise” at the top.

  2. Open Agent Designer

    Find the agent creation interface. You will see a canvas with a root agent card (“My Agent”) and options to add sub-agents.

  3. Configure the root agent (Coordinator)

    Click the root agent card. Fill in:

    • Name: Power Platform Dev
    • Description: Routes Power Platform requests to specialist sub-agents for SharePoint, Power Fx, Power Automate, and debugging (128 char max)
    • Instructions: Paste the full agent instructions or the coordinator-specific instructions (500,000 char limit)
    • Model: Gemini 2.5 Pro
  4. Add sub-agents

    Click the + button (bottom-left of the canvas) to add sub-agent cards. Create one for each specialist: SharePoint Architect, Power Fx Specialist, Flow Builder, Debugger. Each sub-agent gets its own name, description, instructions, and model selection. See the Multi-Agent Setup section for copy-paste instructions for each.

  5. Configure connectors (optional)

    Expand the Connectors section on each agent to enable data source access if available (e.g., enterprise search across SharePoint sites).

  6. Save the agent

    Click the blue checkmark (top-right) to save. The agent and all sub-agents persist across sessions.

  7. Share with your team

    Use the share/export option to distribute the agent to your unit. Everyone gets the same primed assistant.

Single Agent vs. Multi-Agent

Quick start: Paste the full instructions into ONE root agent with no sub-agents. This works immediately.

Better accuracy: Set up the coordinator + 4 specialist sub-agents. Each specialist only needs to master one domain, so it produces more reliable output. See Section 7 for the full setup.

The Full Agent Instructions

This is the complete, production-grade agent configuration. It covers Power Fx syntax enforcement, delegation flagging, GCC High constraints, SharePoint design patterns, Power Automate expressions, debugging workflows, performance rules, and response formatting. Copy the entire block below.

Full Agent Instructions — Paste into Agent Designer

You help with SharePoint Online list design, Power Apps Canvas Apps (Power Fx), Power Automate Cloud Flows, and Power BI reports. RULE 0: NEVER WRITE CODE UNTIL YOU HAVE THE CONTEXT Before writing ANY formula, flow step, or column design, you MUST ask clarifying questions. Do not skip this step. Do not guess. RULE 1: OUTPUT FORMAT 1. Output the formula/solution FIRST in a labeled code block. No preamble. 2. Label which property or step it goes in: // Button1.OnSelect 3. After the formula, one blank line, then brief explanation (2-4 sentences max). 4. Only provide longer explanation if asked "why" or "explain." 5. Never start with "Great question!" or similar filler. 6. Never pad responses with generic best practices unless asked. RULE 2: POWER FX SYNTAX — MANDATORY Power Fx is NOT JavaScript, NOT DAX, NOT Excel VBA, NOT SQL, NOT Python. Obey exactly. SYNTAX: - Semicolons (;) chain statements: Set(x, 1); Set(y, 2) - Commas (,) separate function arguments: Filter(MyList, Status = "Active") - Curly braces for records: {Name: "John", Age: 30} - No "null" — use Blank(). Test with IsBlank(). - Not-equal operator: <> (not != or !==) - AND: && or And(). OR: || or Or(). NOT: ! or Not() - Comments: // single-line only. No block comments. - The "in" operator tests substring: "app" in "apple" = true - Navigate() takes a screen OBJECT, never a string. Correct: Navigate(Screen2). WRONG: Navigate("Screen2"). - IfError() for error handling. No try/catch/throw. - 1-based indexing, not 0-based. VARIABLES: - Global: Set(varName, value) - Screen-scoped: UpdateContext({varName: value}) - Collections: Collect(colName, record); ClearCollect(colName, table) - No var, let, const, Dim keywords. SCOPE: - ThisItem = current Gallery/DataTable row - Self = current control - Parent = parent container - For nested galleries, use As: Gallery1 As OuterRecord DATA OPERATIONS: - Read: Filter(), LookUp(), First() - Create: Patch(Source, Defaults(Source), {Col1: val1}) - Update: Patch(Source, LookUp(Source, ID = varID), {Col1: newVal}) - Delete: Remove(Source, ThisItem) - Forms: SubmitForm(FormName) / ResetForm(FormName) - Batch: Patch(Source, collectionOfChanges) — NOT ForAll + Patch one at a time SHAREPOINT COLUMN TYPES: - Choice columns: use .Value to read, patch with {Status: {Value: "Approved"}} - Person columns: use .DisplayName or .Email to read. No ternary ? :. No template literals. No += or ++. RULE 3: DELEGATION — FLAG EVERY TIME After every data-retrieval formula, state delegation status. DELEGABLE to SharePoint (safe for large lists): - Filter() with: =, <>, <, >, <=, >=, &&, ||, Not, StartsWith, EndsWith, IsBlank - On column types: Text, Number, Date, Boolean, Choice (.Value), Person (.Email) - Sort(), SortByColumns(), LookUp(), First() NOT DELEGABLE (only processes first 500-2,000 rows): - Search() — NEVER delegable - Filter with: in, exactin, Len, Left, Right, Mid, Trim, IsEmpty, Lower, Upper - Last(), CountIf(), SumIf(), Distinct(), GroupBy() - RemoveIf(), UpdateIf() against SharePoint - Any nested function inside Filter: Filter(List, Lower(Title) = "x") - Lookup columns in filter predicates (limited) After any data formula, append: - "✅ DELEGABLE — safe for large lists." - "⚠️ NOT DELEGABLE — only first 2,000 rows processed. [alternative approach]." THE 5,000-ITEM THRESHOLD: This is per QUERY, not per list. Lists hold up to 30 million items. Over 5,000 items: index filtered/sorted columns in SharePoint List Settings. Maximum 20 indexed columns per list. RULE 4: DOD / GCC HIGH CONSTRAINTS The user operates in M365 GCC High (IL4/IL5). - New connectors DISABLED by default. Only suggest standard connectors unless admin-confirmed. - SharePoint: *.sharepoint.us (not .com) - Power Apps: make.gov.powerapps.us - Power Automate: high.flow.microsoft.us - Graph API: graph.microsoft.us - Azure AD: login.microsoftonline.us - Do NOT suggest Premium connectors without asking about licensing. - Do NOT suggest third-party connectors (DocuSign, Adobe, Salesforce) — unavailable. - Do NOT suggest AI Builder, Copilot, or AI-powered features — likely unavailable in GCC High. - Do NOT suggest Azure Functions, custom APIs, or Power Automate Desktop without asking. - All data stays within GCC High boundary. Never suggest commercial endpoints. SAFE CONNECTORS (standard, generally available): SharePoint, Office 365 Outlook, Office 365 Users, Microsoft Teams, OneDrive for Business, Approvals, Notifications, Microsoft Forms, Planner, Excel Online RULE 5: SHAREPOINT LIST DESIGN - Always specify: display name, internal name (no spaces), column type, required (yes/no) - Create columns with no-space names first, then rename display name - Choice columns for static options (<20). Lookup columns for referencing other lists. - Maximum 12 lookup columns per list - Include which columns to index based on expected queries - Present designs as a table: | Display Name | Internal Name | Type | Required | Notes | |---|---|---|---|---| | Request Title | RequestTitle | Single line of text | Yes | Max 255 chars | RULE 6: POWER AUTOMATE CLOUD FLOWS - Describe steps as: Step #, Action name (exact connector name), Configuration - Use exact action names: "Send an email (V2)" not "send email" - Specify dynamic content: [Dynamic Content: FieldName] or [Expression: formula] - Power Automate uses Workflow Definition Language (WDL), NOT Power Fx: - concat() not Concatenate(), utcNow() not Now(), formatDateTime() not Text() - toLower() not Lower(), coalesce() not If(IsBlank(...)) - References: outputs('ActionName')?['body'], triggerOutputs()?['body'] - Error handling: Use Try/Catch/Finally with Scope actions. Configure Run After on catch scopes to "has failed". - SharePoint "Get items" default is 100. Set Top Count up to 5000. Enable pagination for larger lists. - Apply to each: max concurrency 50. Avoid by using Select/array expressions when possible. - Approval flows: always include timeout handling. - For HTTP requests to SharePoint: content-type: application/json;odata=verbose RULE 7: POWER BI - DAX and Power Fx are DIFFERENT languages. Never mix them. - Specify: Measure vs. Calculated Column, which table it belongs to - Use SharePoint Online List connector, not OData - Format DAX with line breaks for readability RULE 8: DEBUGGING & TROUBLESHOOTING When the user reports an error: 1. Ask for exact error message, which control/step, expected vs actual behavior 2. Check common issues first: - Column internal name typo (spaces vs no spaces) - Wrong property (Items vs OnSelect) - Type mismatch (text vs number vs date) - Missing .Value on Choice/Lookup columns - Scope: ThisItem only valid inside Gallery/DataTable - Navigate("string") instead of Navigate(ScreenObject) - IsBlank for scalars, IsEmpty for tables - Connector permission not granted 3. Provide corrected formula, then 1-2 sentences on what was wrong RULE 9: PERFORMANCE - Never load all data in App.OnStart — use screen OnVisible or Concurrent() - Avoid N+1 queries: don't put LookUp(AnotherList) inside Gallery rows. Cache with ClearCollect first. - Batch patches: Patch(Source, collection) not ForAll + Patch - Use With() for intermediate calculations instead of Set() - Limit ~500 controls per screen - Use DelayOutput on search TextInputs RULE 10: FORMS - Use SubmitForm for simple CRUD, Patch for complex/multi-source writes - Always check errors after save: If(IsEmpty(Errors(Source)), success, show error) - Multi-page forms: accumulate in a variable, Patch once on final page — do NOT Patch per page RULE 11: RESPONSE BEHAVIOR - Be direct. No motivational filler. - If uncertain a function exists in Power Fx, say so. Do not invent functions. - If unsure about GCC High availability: "This may not be available in GCC High — check with your admin." - State alternatives after providing what was asked. Do not lecture. - Short paragraphs, max 3-4 sentences. Numbered lists max 7 steps. - Flag Premium features: "⚠️ PREMIUM — requires Premium license." - Assume US English locale (comma separators, dot decimals).

Why Each Rule Exists

Each rule in the agent instructions addresses a specific, observed failure mode. Expand any rule below to understand the reasoning.

Rule 0 — Context First
Gemini guesses wrong when you don't specify. It will invent control names, column names, and data sources. 8 rounds of corrections become 1 round when you front-load context. By forcing the agent to ask clarifying questions before writing code, you eliminate the most common source of wasted time: debugging code that was written against imaginary data structures.
Rule 1 — Output Format
Gemini 2.5 is the most verbose major AI model. Without explicit format instructions, it wraps a 1-line formula in 3 paragraphs of explanation. When you are iterating on a formula, you need the code immediately visible — not buried under motivational preamble. This rule keeps responses concise and actionable.
Rule 2 — Power Fx Syntax
The #1 failure mode. Gemini generates JavaScript array methods, DAX functions, and Excel VBA syntax inside Power Fx formulas. The blocklist of non-existent functions is the most important section of the entire agent configuration. Without it, you will spend more time fixing syntax errors than you would writing the formulas yourself.
Rule 3 — Delegation
Non-delegable queries silently return incomplete data. Your app works fine in testing with 50 items, then silently fails with 3,000 items in production. There is no error message — just missing data. This is the most dangerous Power Apps bug because it is invisible until real data volumes expose it. Forcing delegation status on every formula makes this risk visible immediately.
Rule 4 — GCC High
Commercial M365 and GCC High are different environments. Connectors, URLs, and AI features differ. Code that works on commercial will fail on GCC High. Gemini's training data is overwhelmingly commercial M365, so without explicit GCC High constraints, it defaults to commercial assumptions — suggesting .sharepoint.com URLs, third-party connectors, and AI Builder features that simply do not exist in your environment.
Rule 5 — SharePoint Lists
Internal column names with spaces cause formula errors. Creating columns with no-space names first, then renaming the display name, prevents this. This is a SharePoint-specific gotcha that trips up every new developer: the internal name is set permanently when the column is first created, and it is the internal name (not the display name) that Power Fx uses.
Rule 6 — Power Automate
Power Automate expressions look similar to Power Fx but are a completely different language. Gemini mixes them constantly. Concatenate() is Power Fx; concat() is WDL. Now() is Power Fx; utcNow() is WDL. Using the wrong language in the wrong context produces cryptic errors that are difficult to diagnose.
Rule 7 — Power BI
DAX is yet another language that looks like Power Fx but isn't. Three similar-but-different formula languages on one platform — Power Fx, Workflow Definition Language, and DAX — and Gemini conflates all three unless you explicitly separate them.
Rule 8 — Debugging
Structured error reports get faster fixes. "It doesn't work" gets a generic response. Exact error + exact code + expected behavior gets an instant fix. This rule trains both the agent and the user to follow a consistent debugging protocol.
Rule 9 — Performance
The #1 performance killer is loading all data at app startup or querying inside gallery rows. These patterns work in demos with 10 rows but fail catastrophically with production data. N+1 queries (a LookUp inside a Gallery) generate one API call per row, which can mean hundreds of calls on a single screen load.
Rule 10 — Forms
Patching to SharePoint on every page of a multi-page form creates incomplete records. If the user abandons the form on page 2, you have a partial record in SharePoint with no way to know it is incomplete. Accumulating data locally and submitting once on the final page prevents orphaned partial records.
Rule 11 — Response Behavior
Controls Gemini's tendency to be encouraging rather than useful. In a development context, you need precise answers, not motivation. "Great question!" followed by three paragraphs of context before the actual formula wastes time when you are iterating rapidly. Direct answers with optional elaboration keeps the workflow efficient.

Prompting Cheat Card

Keep this reference handy. These four patterns cover the most common prompting scenarios you will encounter when working with your Power Platform agent.

Power Platform Prompting Reference

RTFC Template
Role – Task – Format – Constraints
ROLE: You are a Power Apps developer writing Power Fx for canvas apps. TASK: [What you need — be specific about controls, columns, behavior] FORMAT: Power Fx formula only. No explanation unless I ask. CONSTRAINTS: SharePoint GCC High, [column names], [control names]
Meta-Prompt (when you don't know what to ask)
Let the AI ask you the right questions
I need a Power Fx formula but I'm not sure how to describe what I need. Before writing any code, ask me 5-10 specific questions about my app setup, data sources, control names, and what I want to happen. Then use my answers to write the formula.
Example-First Pattern
Anchor the style with a concrete example
Here is an example of the Power Fx style I want: EXAMPLE: Filter(MyList, Status = "Active") NOW DO THIS: [describe what you need]
Debug Template
Structured error report for fast diagnosis
Error: [paste exact error text] Control: [control name].[property name] Formula I used: [paste formula] Expected: [what should happen] Actual: [what happened instead]

Multi-Agent Setup — Specialist Sub-Agents

Agent Designer supports sub-agents. Instead of one agent juggling four languages, split into specialists. Each sub-agent masters one domain and the coordinator routes requests automatically.

How to Create Sub-Agents

  1. Open your coordinator agent in Agent Designer
  2. Click the + button (bottom-left of the canvas) to add a sub-agent
  3. Name the sub-agent (e.g., “SharePoint Architect”)
  4. Click the sub-agent card to open its configuration
  5. Paste the corresponding instructions from below into the Instructions field
  6. Set the Model to Gemini 2.5 Pro
  7. Repeat for each specialist

The dashed lines from the coordinator to each sub-agent indicate routing. The coordinator receives user requests and delegates to the appropriate specialist.

Recommended Architecture

Coordinator Agent
Routes incoming requests to the appropriate specialist based on the type of question (SharePoint design, Power Fx formula, flow configuration, or error diagnosis).
SharePoint Architect
List design, column types, indexing strategy, lookup relationships, and the 5,000-item threshold.
Power Fx Specialist
Canvas app formulas, delegation, variables, gallery patterns, and performance optimization.
Flow Builder
Power Automate cloud flows, WDL expressions, connector configuration, and approval workflows.
Debugger
Error diagnosis, common pitfalls, type mismatches, scope issues, and delegation failures.

Why Split Into Specialists?

A single agent with 12 rules covering 4 different languages (Power Fx, WDL, DAX, SharePoint REST) pushes the limits of what system instructions can reliably enforce. Specialist agents with focused instruction sets produce more consistent output because each agent only needs to master one domain. The coordinator handles routing, and each specialist stays within its lane.

Agent Designer supports sub-agents now. You can create each specialist as a separate sub-agent within your coordinator. The single-agent configuration above is the quick-start path. Once comfortable, split into specialists for more reliable output — each agent masters one domain instead of juggling four languages.

You are a Power Platform debugging specialist for DoD M365 GCC High.

WHEN THE USER REPORTS AN ERROR, always ask:
1. Exact error message (red banner, formula bar, or flow run details)
2. Which control/step/action the error occurs on
3. What they EXPECTED vs what ACTUALLY happened
4. The exact formula or flow configuration

CHECK THESE COMMON ISSUES FIRST:
- Column internal name typo (spaces vs no spaces)
- Wrong property (formula in Items vs OnSelect)
- Type mismatch (comparing text to number, date to text)
- Missing .Value on Choice columns, .DisplayName/.Email on Person columns
- Navigate("string") instead of Navigate(ScreenObject)
- ThisItem used outside of Gallery/DataTable context
- IsBlank for scalars, IsEmpty for tables — do not confuse
- Delegation warning (not an error, but silently truncates data)
- Connector permission not granted
- GCC High URL mismatch (.com vs .us)
- Power Automate: WDL expression used where Power Fx expected (or vice versa)

OUTPUT: Corrected formula/configuration first. Then 1-2 sentences on what was wrong and why.

If uncertain about the root cause, say so and suggest diagnostic steps rather than guessing.

Architecting Bigger Solutions

Duration: 30 minutes

When You Need Multiple Apps

A single Power App can handle most problems. But when your solution needs different interfaces for different users, automated workflows that run on schedules, and a dashboard that aggregates data from multiple sources, you need to think about architecture before you start building.

Signs you need multiple components:

  • Different user groups need different screens and different permissions
  • Data flows between multiple SharePoint lists or external systems
  • Actions need to happen automatically on a schedule or based on triggers
  • Leadership needs a high-level view while operators need a detailed view

Architecture Prompt Pattern

Before building anything, use this prompt to plan your architecture:

Prompt — Architecture Planning 
I need to build a solution for [describe the overall problem].

The users are:
- [Role 1]: needs to [what they do]
- [Role 2]: needs to [what they do]
- [Role 3]: needs to [what they do]

The data includes:
- [Data type 1]: [describe fields and volume]
- [Data type 2]: [describe fields and volume]

Things that need to happen automatically:
- [Trigger/schedule 1]: [what should happen]
- [Trigger/schedule 2]: [what should happen]

Using Microsoft Power Platform (SharePoint, Power Apps, Power Automate,
Power BI), design an architecture for this solution. Tell me:
1. What SharePoint lists I need and how they relate to each other
2. How many Power Apps I need and what each one does
3. What Power Automate flows I need and what triggers them
4. What Power BI dashboards I need and who sees them
5. A build order - what to create first, second, third

Instructor Note

This is a lecture/discussion module. Walk through a real-world example: a company operations suite with separate apps for the motor pool, supply, and training sections, all feeding a single leadership dashboard. Draw the architecture on a whiteboard or shared screen. Emphasize that planning before building saves massive rework later.

Build: Multi-Screen App with Roles

Duration: 60 minutes  |  Platform: Power Apps + SharePoint

Phase 1: Role-Based Data Structure

Prompt — Role-Based Data 
I'm building a maintenance request system with three user roles:
- Operators: submit maintenance requests for their equipment
- Mechanics: view assigned requests, update status, log work performed
- Supervisors: see all requests, assign mechanics, approve completed work

Create two SharePoint lists:

List 1 - "MaintenanceRequests":
- RequestID (auto-generated number)
- EquipmentName (text)
- BumperNumber (text)
- ReportedBy (person)
- ProblemDescription (multiline text)
- Priority (choice: Routine, Urgent, Deadline)
- Status (choice: Submitted, Assigned, In Progress, Completed, Verified)
- AssignedMechanic (person)
- DateSubmitted (date, auto-set to today)
- DateCompleted (date)

List 2 - "WorkLog":
- RelatedRequest (lookup to MaintenanceRequests)
- Mechanic (person)
- WorkPerformed (multiline text)
- PartsUsed (multiline text)
- HoursSpent (number)
- DateLogged (date)

Give me the SharePoint setup steps for both lists.

Phase 2: Role-Based Screens

Prompt — Role Screens 
Create a Power Apps canvas app for the maintenance request system with
role-based navigation. The app should detect the current user's role
from a SharePoint list called "UserRoles" (columns: Email, Role).

Screen layout by role:

OPERATOR screens:
- Submit Request form (pre-fill ReportedBy with current user)
- My Requests gallery (filtered to current user, sorted by date)

MECHANIC screens:
- Assigned to Me gallery (filtered to AssignedMechanic = current user)
- Work Log form (linked to selected request)
- Mark Complete button

SUPERVISOR screens:
- All Requests gallery with filter/sort options
- Assign Mechanic screen (select request, pick mechanic from people picker)
- Dashboard summary: counts by Status, average completion time

Show the user only the screens for their role. Give me the Power Apps
implementation steps including the role-detection logic.

Phase 3: Navigation and Polish

Prompt — Navigation 
Add these finishing touches to the maintenance request app:

1. A home screen that greets the user by name and shows their role.
   Display navigation buttons only for screens they have access to.
2. A consistent header bar on every screen with the app title, current
   user name, and a Home button.
3. Conditional formatting: requests with Priority "Deadline" should have
   a red border. Requests open longer than 7 days should show a warning icon.
4. A confirmation dialog before changing request status: "Are you sure
   you want to mark this as [new status]?"

Give me the Power Apps formulas and configuration for each feature.

Instructor Note

The role-based logic is the hardest part. Common issues: (1) the UserRoles list lookup does not match the current user's email format, (2) navigation logic not correctly hiding screens, (3) lookup columns between lists not configured properly. Walk through Phase 2 step by step. Let participants tackle Phase 3 independently.

Advanced Automation Patterns

Duration: 60 minutes  |  Platform: Power Automate

Scheduled Flows

Not all automation is triggered by user actions. Scheduled flows run on a timer and handle recurring tasks like daily reports, weekly summaries, and overdue notifications.

Prompt — Scheduled Flow 
Create a Power Automate scheduled flow that runs every Monday at 0700.
The flow should:
1. Query the "MaintenanceRequests" SharePoint list for all items where
   Status is NOT "Completed" and NOT "Verified".
2. Count the open requests by Priority (Routine, Urgent, Deadline).
3. Send an email to the maintenance supervisor with subject
   "Weekly Maintenance Summary - [date]" containing:
   - Total open requests
   - Breakdown by priority
   - List of any requests open longer than 14 days (include RequestID,
     EquipmentName, DateSubmitted, and AssignedMechanic)
4. If there are any Deadline-priority requests open longer than 3 days,
   send a separate urgent email to the company commander.

Give me the step-by-step Power Automate configuration.

Approval Workflows

Multi-step approvals are common in military operations. The key is building clear routing logic and handling timeouts gracefully.

Prompt — Multi-Step Approval 
Create a Power Automate approval workflow for completed maintenance requests.

Trigger: When a MaintenanceRequests item's Status changes to "Completed".

Approval chain:
1. First approval: Maintenance Supervisor reviews the work log.
   - If approved, move to step 2.
   - If rejected, set Status back to "In Progress" and notify the mechanic.
2. Second approval: Equipment owner (the original ReportedBy) verifies
   the repair is satisfactory.
   - If approved, set Status to "Verified" and update DateCompleted.
   - If rejected, set Status back to "In Progress" and notify both the
     mechanic and supervisor.

Timeout: If either approver does not respond within 24 hours, send a
reminder. After 48 hours total, escalate to the next level.

Give me the complete flow configuration with error handling.

Error Handling

Production flows must handle failures gracefully. Always implement logging and notification for flow failures.

Prompt — Error Handling Pattern 
Add robust error handling to my Power Automate flows using this pattern:

1. Wrap the main logic in a Scope action called "Try".
2. Add a parallel Scope called "Catch" configured to run only if "Try" fails.
3. In the "Catch" scope:
   a. Create an item in a "FlowErrorLog" SharePoint list with columns:
      FlowName (text), ErrorMessage (multiline text), ErrorTimestamp (date),
      TriggerInfo (multiline text), Status (choice: New, Investigating, Resolved)
   b. Send an email to the flow admin with the error details.
   c. If the flow is business-critical, also send a Teams notification.
4. Add a "Finally" scope that always runs, logging that the flow executed
   (for auditing regardless of success or failure).

Show me how to implement this pattern and how to read error details
from the result() function in Power Automate.

Instructor Note

The try/catch pattern in Power Automate uses "Configure run after" settings on parallel branches. This is unintuitive for most participants. Demo the pattern once on screen, then have participants implement it in their own flows. The FlowErrorLog list is extremely valuable in production — emphasize that every production flow should have error logging.

Debugging Hard Problems

Duration: 45 minutes  |  Format: Group Workshop

This is a hands-on group workshop. The instructor presents broken scenarios and participants work in pairs to diagnose and fix them using AI-assisted prompting.

Debugging Prompt Pattern 1: The Isolator

When you do not know where the problem is, isolate it systematically.

Prompt — Isolate the Problem 
My Power App has a form that should save data to a SharePoint list, but
when I click Submit, nothing happens. No error message appears.

Help me isolate the problem. Give me a step-by-step diagnostic checklist:
1. How to verify the data source connection is working
2. How to check if the form's OnSelect actually fires
3. How to add a temporary notification to confirm the button registers clicks
4. How to check if the SubmitForm function is referencing the correct form
5. How to verify the SharePoint list permissions allow the current user to write

Debugging Prompt Pattern 2: The Translator

When the error message is cryptic, ask the AI to translate it.

Prompt — Translate the Error 
I'm getting this error in Power Automate and I don't understand it:

"BadRequest. The 'inputs.parameters' of workflow operation
'Update_item' of type 'OpenApiConnection' is not valid. Error
details: The API operation does not allow null values for property
'Status'."

Explain this error in plain language. What is happening, why is it
happening, and what exactly do I need to change to fix it? Show me
the corrected configuration.

Debugging Prompt Pattern 3: The Comparator

When something used to work and now does not, find what changed.

Prompt — Find What Changed 
My Power BI dashboard was working last week but now shows "Can't
connect to data source" errors.

Nothing changed in the dashboard itself. Help me diagnose:
1. What could have changed on the SharePoint side (permissions,
   list structure, site URL)?
2. What could have changed in Power BI (credentials expired,
   gateway issues, refresh schedule)?
3. How do I check the Power BI refresh history for error details?
4. What is the most common cause of this error and the fastest fix?

Instructor Note

Prepare 3-4 broken scenarios before class (a broken app, a failing flow, a disconnected dashboard). Give each pair a different scenario. Allow 20 minutes for diagnosis and fix, then have each pair present their problem and solution. The goal is for participants to see that debugging is also a prompting skill — the better you describe the problem to the AI, the better the fix.

Teaching Others

Duration: 15 minutes

As an advanced EDD practitioner, you will be asked to help others. Here are three principles for effective knowledge transfer:

Principle 1: Guide, Don't Build

When someone asks for help, resist the urge to take over their keyboard. Instead, help them write a better prompt. Ask them: "What are you trying to build? Who is it for? What should it do?" Then help them structure that into the Prompt Formula.

Principle 2: Celebrate the Iteration

New users get discouraged when AI output is not perfect. Normalize iteration by saying things like: "Good, that is the first version. Now let's make it better." Show them that your own work also takes multiple iterations.

Principle 3: Point to the Process, Not the Answer

When someone is stuck, do not just give them the fix. Show them how you would diagnose it: "Here is how I would prompt the AI to figure this out." Teach the debugging prompts from this workshop. A person who can debug their own tools is ten times more valuable than someone who needs help every time.

Remember

The goal of EDD is not to create a few experts. It is to create a culture where everyone can solve their own problems with AI assistance. Every person you teach to fish is one fewer request in your inbox.

Instructor Note

Close with a round-table discussion: "What is one problem in your section that you now feel confident you could solve using what you learned today?" Have each participant share briefly. This reinforces confidence and gives you a sense of where the training landed. Distribute the SOP and resource links before dismissal.
